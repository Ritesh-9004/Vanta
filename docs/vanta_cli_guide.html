Build order — what to ship first

SHIP FIRST
vanta search
Search the registry from terminal. Calls your existing API. Shows results with quality scores. Easiest to build, immediately useful.
Time: 1 day
SHIP FIRST
vanta install
Downloads package from registry, extracts to project folder, updates vanta.toml and vanta.lock. The core feature.
Time: 3-4 days
SHIP FIRST
vanta init
Creates vanta.toml in current directory. Asks: board, framework. Generates starter config. First command every new user runs.
Time: half a day
V1.1
vanta add
Adds a dependency to vanta.toml without downloading. Like npm install --save vs just editing package.json.
Time: 1 day
V1.1
vanta list
Shows all installed packages with versions and quality scores. Reads from vanta.lock.
Time: half a day
V1.1
vanta update
Updates packages to latest versions within semver constraints. Rewrites vanta.lock.
Time: 1-2 days
V2
vanta build
Wraps CMake + Ninja. Compiles firmware. Shows memory report. Hardest command — skip for v1.
Time: 2-3 weeks
V2
vanta flash
Wraps esptool. Flashes firmware to device. Depends on vanta build working first.
Time: 1 week
V2
vanta publish
Publishes a library to the registry. Needs auth system first.
Time: 1 week
The v1 goal: vanta init + vanta search + vanta install. That's it. A developer can find a library and add it to their project. Everything else comes after you have real users.

package.json for the CLI

apps/cli/package.json
{ "name": "vanta", "version": "0.1.0", "description": "Universal MCU package manager", "bin": { "vanta": "./src/index.ts" // this makes 'vanta' command work }, "scripts": { "dev": "bun src/index.ts", "build": "bun build src/index.ts --compile --outfile vanta", "build:win": "bun build src/index.ts --compile --target=bun-windows-x64 --outfile vanta.exe" } }
bun build --compile creates a single executable binary — no Node.js or Bun runtime needed on the user's machine. Windows users get vanta.exe, Mac/Linux get vanta. This is how you distribute the CLI.

Core Files — Exact Code
The key files to write first

src/index.ts — entry point

src/index.ts
#!/usr/bin/env bun import { Command } from 'commander' import { searchCommand } from './commands/search' import { installCommand } from './commands/install' import { initCommand } from './commands/init' import { listCommand } from './commands/list' const program = new Command() program .name('vanta') .description('Universal MCU package manager') .version('0.1.0') program .command('init') .description('Initialize a new vanta project') .action(initCommand) program .command('search <query>') .description('Search the vanta registry') .option('-p, --platform <platform>', 'Filter by platform (esp32, stm32, rp2040)') .option('-f, --framework <framework>', 'Filter by framework (arduino, espidf)') .action(searchCommand) program .command('install [packages...]') .alias('i') .description('Install packages') .action(installCommand) program .command('list') .alias('ls') .description('List installed packages') .action(listCommand) program.parse()

src/core/config.ts — reads/writes vanta.toml

src/core/config.ts
import { readFileSync, writeFileSync, existsSync } from 'fs' import { join } from 'path' import TOML from '@iarna/toml' // bun add @iarna/toml export interface VantaConfig { project: { name: string version: string } target: { board: string chip: string framework: string } dependencies: Record<string, string> } export function readConfig(cwd = process.cwd()): VantaConfig | null { const configPath = join(cwd, 'vanta.toml') if (!existsSync(configPath)) return null const content = readFileSync(configPath, 'utf-8') return TOML.parse(content) as VantaConfig } export function writeConfig(config: VantaConfig, cwd = process.cwd()) { const configPath = join(cwd, 'vanta.toml') writeFileSync(configPath, TOML.stringify(config as any)) } export function addDependency(name: string, version: string) { const config = readConfig() if (!config) throw new Error('No vanta.toml found. Run vanta init first.') config.dependencies[name] = `^${version}` writeConfig(config) }

src/core/registry.ts — calls your API

src/core/registry.ts
const REGISTRY_URL = 'https://registry.vanta.dev' // your API URL export async function searchPackages(query: string, filters?: { platform?: string framework?: string }) { const params = new URLSearchParams({ q: query, ...filters }) const res = await fetch(`${REGISTRY_URL}/search?${params}`) if (!res.ok) throw new Error('Registry unreachable') return res.json() } export async function getPackage(name: string, version?: string) { const url = version ? `${REGISTRY_URL}/packages/${name}/${version}` : `${REGISTRY_URL}/packages/${name}` const res = await fetch(url) if (res.status === 404) throw new Error(`Package "${name}" not found in registry`) return res.json() } export async function downloadTarball(tarballUrl: string, destPath: string) { const res = await fetch(tarballUrl) const buffer = await res.arrayBuffer() await Bun.write(destPath, buffer) }

What vanta install Does — Step by Step
Exactly what happens on disk when a user runs vanta install bme280
1
Read vanta.toml
Find the project root by walking up directories looking for vanta.toml. Read target chip and framework — needed for compatibility checking.
// Find project root const config = readConfig() // reads vanta.toml // config.target.chip = "esp32s3" // config.target.framework = "arduino"
2
Resolve version from registry
Call GET /packages/bme280 on your API. Get the latest version that satisfies the semver range. Also fetch its dependencies list.
const pkg = await getPackage('bme280') // pkg.version = "2.4.1" // pkg.tarball_url = "https://r2.../bme280-2.4.1.tar.gz" // pkg.dependencies = { "ArduinoJSON": "^7.0.0" }
3
Check compatibility
Does this package support the user's chip and framework? Check pkg.targets includes "esp32s3" and pkg.frameworks includes "arduino". Warn if not compatible.
4
Resolve transitive dependencies
bme280 depends on ArduinoJSON@^7.0.0. Fetch that too. Check if it's already installed. Build the full install list before downloading anything.
// Install queue: [bme280@2.4.1, ArduinoJSON@7.0.4] // ArduinoJSON already installed? skip it
5
Download tarballs
Download each package tarball from Cloudflare R2. Verify SHA256 hash. Save to .vanta/packages/bme280@2.4.1.tar.gz
6
Extract to .vanta/packages/
Extract tarball into .vanta/packages/bme280@2.4.1/ folder. This contains the library's src/, include/, and CMakeLists.txt.
.vanta/ └── packages/ ├── bme280@2.4.1/ │ ├── src/bme280.c │ ├── include/bme280.h │ └── CMakeLists.txt └── ArduinoJSON@7.0.4/ └── src/ArduinoJSON.h
7
Update vanta.toml + vanta.lock
Add bme280 = "^2.4.1" to [dependencies] in vanta.toml. Write exact resolved versions to vanta.lock for reproducible builds.
# vanta.lock (auto-generated, always commit this) [packages] bme280 = { version = "2.4.1", sha256 = "abc123..." } ArduinoJSON = { version = "7.0.4", sha256 = "def456..." }
8
Show success + usage hint
Tell the user what was installed and how to use it in their code. This is the UX that makes people smile.
✓ 2 packages installed In your code: #include "bme280.h"
For v1 — skip CMake generation. Don't try to auto-configure the build system yet. Just download and extract the libraries. Tell users to add the include path manually. CMake auto-generation comes in v2 when you have real users telling you they need it.

# Vanta CLI — Context for AI Agent I am building the CLI for Vanta — a universal MCU package manager. The CLI is a TypeScript app built with Bun, using Commander for argument parsing. ## What the CLI does vanta init → creates vanta.toml in current directory vanta search → queries registry API, shows results in terminal vanta install → downloads packages, extracts to .vanta/, updates toml+lock vanta list → shows installed packages from vanta.lock vanta add → adds dep to vanta.toml without downloading ## File structure apps/cli/ ├── src/index.ts ← entry, Commander setup ├── src/commands/ ← one file per command ├── src/core/config.ts ← read/write vanta.toml ├── src/core/lockfile.ts ← read/write vanta.lock ├── src/core/registry.ts ← HTTP calls to registry API ├── src/core/downloader.ts ← download + extract tarballs └── src/utils/logger.ts ← chalk colored output ## Key files in user's project vanta.toml ← user writes this (board, framework, deps) vanta.lock ← auto-generated (exact versions + sha256) .vanta/ ← git-ignored, contains extracted packages ## Registry API base URL https://registry.vanta.dev (or localhost:3001 in dev) Endpoints: GET /search?q=bme280&platform=esp32&framework=arduino GET /packages/:name GET /packages/:name/:version ## Dependencies installed commander, chalk, ora, @inquirer/prompts, semver, tar, @iarna/toml ## vanta.toml format [project] name = "my_project" version = "0.1.0" [target] board = "esp32-devkitc-v4" chip = "esp32s3" framework = "arduino" [dependencies] bme280 = "^2.4.1" ## vanta.lock format [packages] bme280 = { version = "2.4.1", sha256 = "abc...", url = "https://..." } ## What install does (in order) 1. Read vanta.toml for target chip + framework 2. Call GET /packages/:name to resolve latest version 3. Check compatibility (pkg.targets includes chip) 4. Resolve transitive dependencies recursively 5. Download tarballs to temp, verify SHA256 6. Extract to .vanta/packages/name@version/ 7. Update vanta.toml [dependencies] 8. Write vanta.lock with exact versions 9. Print success + #include hint ## Rules - Error messages must be actionable: BAD: "Error: 404" GOOD: 'Package "bme280" not found. Did you mean "bme280_idf"?' - Always show a spinner (ora) during network calls - Use chalk.green for success, chalk.red for errors, chalk.yellow for warnings - Never crash silently — always catch and explain errors - v1 does NOT do CMake generation — just download + extract
